# WordPress to Astro 迁移设计

## 目标

将当前线上 WordPress 博客替换为本仓库的 Astro 博客，只保留文章内容的迁移。

## 范围

### 保留

- 文章标题
- 文章正文
- 发布时间
- 文章封面图

### 不保留

- 评论
- 分类页
- 标签页
- 旧独立页面
- WordPress 账号体系
- 站内搜索索引的历史数据

## 现状

- 线上站点当前运行在云服务器上，前端由 Nginx 提供服务。
- 当前线上内容来自 WordPress。
- 本地仓库是 Astro 博客项目，适合导出为静态站点后替换线上站点。

## 设计方案

### 1. 内容迁移

从 WordPress 导出文章内容，转换为 Astro 可直接消费的 `md/mdx` 文件。

每篇文章统一落盘为：

- `src/content/blog/<slug>/post.mdx`
- 如有英文镜像，再保留为独立文件；本次迁移默认不新增英文镜像

文章 Frontmatter 仅保留：

- `title`
- `description`
- `publishDate`
- `heroImage`
- `draft`

### 2. 站点接管

Astro 构建产物作为新的线上站点内容，Nginx 由原 WordPress 根目录切换到新的构建目录。

切换原则：

- 先保留现有 WordPress 站点备份
- 再上线 Astro
- 最后验证首页、文章页和静态资源

### 3. 旧链接处理

不做完整的 WordPress URL 兼容层。

最低限度保留：

- 404 页面
- 文章页可访问

如果旧链接量很小，可接受少量 404；如后续发现高频旧链接，再补定向规则。

## 数据流

1. WordPress 导出文章
2. 转成 Astro 文章文件
3. Astro 构建生成静态站点
4. Nginx 切换到新站点目录
5. 公网访问验证

## 风险与约束

- WordPress 导出的 HTML 可能包含短代码、内嵌样式和旧媒体链接，需要人工清理。
- 旧站点如果依赖 WordPress 的动态能力，迁移后会失效。
- 线上切换前必须保留回滚点。

## 验收标准

- 首页可以正常打开
- 文章列表可用
- 文章详情页可用
- 静态资源正常加载
- 旧 WordPress 站点已被替换，但有可回退备份

## 后续实施顺序

1. 导出 WordPress 文章
2. 清洗并转换文章为 Astro 内容
3. 本地构建并预览
4. 在云服务器上准备新站点目录
5. 切换 Nginx
6. 公网验收
